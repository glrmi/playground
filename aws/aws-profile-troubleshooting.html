InvalidClientTokenId부터 프로파일 누락 진단까지 — AWS CLI 인증 문제 점검 가이드

2026-07-02 · Windows / macOS

개요

AWS CLI를 사용하다 보면 특정 프로파일로 명령을 실행했을 때 다음과 같은 인증 오류를 만날 수 있다.

An error occurred (InvalidClientTokenId) when calling the AssumeRole operation:
The security token included in the request is invalid.

처음 보면 AWS 서비스 자체 문제처럼 보일 수 있지만, 실제로는 로컬에 저장된 AWS CLI 프로파일, Access Key, Session Token, source_profile 설정 문제인 경우가 많다.

이번 글에서는 AWS CLI 인증 문제가 발생했을 때 어떤 순서로 확인하면 좋은지 정리한다.

01 AWS CLI 설정 파일 구조

AWS CLI는 기본적으로 두 개의 파일을 사용한다.

설정 파일
config

리전, 출력 형식, Role ARN, source_profile 등 동작 설정을 저장한다. 구성 방식에 따라 민감 정보가 포함될 수 있으므로 외부 공유 전 확인이 필요하다.

~/.aws/config
인증 파일
credentials

Access Key ID, Secret Access Key, Session Token 등 실제 자격증명을 저장한다. 외부 노출 금지.

~/.aws/credentials
프로파일 표기 방식 차이

config 파일은 [profile example], credentials 파일은 [example] 형식을 사용한다. profile 접두어 유무에 주의.

파일 예시
# ~/.aws/config
[profile example]
region = ap-northeast-2
output = json

# ~/.aws/credentials
[example]                    # profile 접두어 없음
aws_access_key_id     = AKIA...
aws_secret_access_key = ...

02 먼저 실행할 진단 명령어

인증 문제가 발생하면 가장 먼저 아래 명령어를 실행한다.

$ aws sts get-caller-identity --profile <profile-name>

# 정상일 때
{
  "UserId": "AROAXXXXXXXXXX:session",
  "Account": "123456789012",
  "Arn": "arn:aws:sts::123456789012:assumed-role/ExampleRole/session"
}

# 실패할 때
An error occurred (InvalidClientTokenId) ...

실패한다면 자격증명, Session Token, Role Assume 설정, 환경변수 중 하나를 의심해야 한다.

Tip

특정 AWS 서비스 명령이 실패했더라도, 먼저 sts get-caller-identity로 인증 자체가 되는지 확인하는 것이 좋다.

03 프로파일 목록과 설정 확인

# 등록된 프로파일 전체 목록
$ aws configure list-profiles

# 특정 프로파일 설정 상세 확인
$ aws configure list --profile <profile-name>

확인할 항목:

  • 어떤 Access Key를 사용하고 있는지
  • 리전이 설정되어 있는지
  • 환경변수에서 값을 가져오고 있는지
  • source_profile을 사용하는 AssumeRole 구조인지

04 AssumeRole 프로파일의 source_profile 확인

AWS CLI 프로파일 중에는 직접 Access Key를 가지지 않고, 다른 프로파일을 기반으로 Role을 Assume하는 경우가 있다.

~/.aws/config
[profile workload-admin]
region            = ap-northeast-2
role_arn          = arn:aws:iam::123456789012:role/ExampleRole
source_profile    = default
role_session_name = workload-admin-session

이 경우 workload-admin은 직접 Access Key를 사용하는 것이 아니라, default 프로파일의 자격증명을 사용해 Role을 Assume한다.

중요

이 구조에서는 credentials 파일에 [workload-admin] 블록이 없어도 정상이다.

실제 Access Key는 source_profile로 지정된 default 프로파일에만 있으면 된다.

workload-admin에서 InvalidClientTokenId가 발생했다면, workload-admin 자체보다 먼저 source_profile인 default를 확인해야 한다.

# source_profile 먼저 확인
$ aws sts get-caller-identity --profile default

# 이후 대상 프로파일 확인
$ aws sts get-caller-identity --profile workload-admin

05 자주 발생하는 에러와 원인

에러 메시지주요 원인확인할 것
InvalidClientTokenId 토큰 또는 자격증명 무효. AssumeRole 중 발생했다면 대상 프로파일보다 source_profile의 자격증명을 먼저 확인한다. source_profile의 Access Key / Session Token
InvalidAccessKeyId Access Key가 AWS에 존재하지 않음 credentials 파일의 Access Key
ExpiredTokenException 임시 토큰 만료 aws_session_token 갱신
NoCredentialProviders 자격증명 없음 프로파일 존재 여부, credentials 파일
config profile could not be found config/credentials에 해당 프로파일 없음 프로파일명 오타, 파일 위치

06 credentials 파일 직접 확인

프로파일 문제가 의심되면 credentials 파일을 직접 확인한다.

cmd / PowerShell
> type %USERPROFILE%\.aws\credentials
> notepad %USERPROFILE%\.aws\credentials
Terminal
# 내용 출력
$ cat ~/.aws/credentials
$ cat ~/.aws/config

# 편집
$ vi ~/.aws/credentials
$ code ~/.aws/credentials   # VS Code 사용 시

파일 구조 예시:

~/.aws/credentials
[default]
aws_access_key_id     = AKIA...
aws_secret_access_key = ...

[workload]
aws_access_key_id     = AKIA...
aws_secret_access_key = ...
aws_session_token     = ...    # 있으면 임시 자격증명 → 만료 가능
aws_session_token 주의

aws_session_token이 있으면 임시 자격증명이다.

임시 자격증명은 수 시간~수일 내 만료될 수 있으므로, 인증 오류가 발생했을 때 먼저 확인해야 한다.

장기 Access Key로 전환할 때는 credentials 파일에서 기존 aws_session_token 항목을 직접 제거해야 할 수 있다.

07 환경변수 확인

credentials 파일을 수정했는데도 계속 같은 오류가 발생한다면 환경변수를 확인해야 한다.

AWS CLI는 환경변수, 프로파일 설정 파일, 명령줄 옵션 등 여러 위치에서 설정값을 읽는다. 환경변수에 오래된 자격증명이 남아 있으면, credentials 파일을 정상으로 수정해도 인증 오류가 계속 발생할 수 있다.

cmd
> set AWS
PowerShell
> Get-ChildItem Env:AWS*
Terminal
$ env | grep ^AWS_
확인할 주요 환경변수
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
AWS_SESSION_TOKEN           # 만료된 값이 남아있으면 파일 수정해도 계속 실패
AWS_PROFILE
AWS_DEFAULT_PROFILE
AWS_SHARED_CREDENTIALS_FILE
AWS_CONFIG_FILE

08 프로파일 재등록

장기 Access Key 기반 프로파일을 등록하거나 갱신할 때는 다음 명령어를 사용한다.

$ aws configure --profile <profile-name>

AWS Access Key ID [None]:        AKIA...
AWS Secret Access Key [None]:    xxxxxxxx
Default region name [None]:      ap-northeast-2
Default output format [None]:    json

# 등록 후 반드시 인증 확인
$ aws sts get-caller-identity --profile <profile-name>
참고

AssumeRole 프로파일을 구성하는 경우에는 Access Key 등록 외에도 ~/.aws/config 파일에 role_arn, source_profile 설정이 필요하다.

임시 자격증명 기반 프로파일을 사용 중이었다면 기존 aws_session_token이 남아 있지 않은지 함께 확인한다.

09 추천 점검 순서

  • 1aws sts get-caller-identity --profile <name> 실행
  • 2aws configure list --profile <name>로 설정 확인
  • 3~/.aws/config에서 role_arn, source_profile 확인
  • 4source_profile의 인증 상태 별도 확인
  • 5~/.aws/credentials에서 Access Key / Session Token 확인
  • 6환경변수 AWS_* 확인
  • 7필요 시 aws configure --profile <name>으로 재등록
  • 8다시 sts get-caller-identity로 검증

정리

AWS CLI 인증 오류는 대부분 다음 중 하나에서 발생한다.

원인 체크리스트

프로파일명 오타 · credentials 파일에 자격증명 없음 · Access Key 삭제 또는 비활성화 · Session Token 만료 · AssumeRole의 source_profile 무효 · 환경변수에 오래된 자격증명 잔류

특정 AWS 서비스 명령이 실패했더라도 바로 서비스 문제로 보기보다는, 먼저 아래 명령으로 인증 상태를 확인하는 것이 좋다.

$ aws sts get-caller-identity --profile <profile-name>

인증이 되면 AWS 서비스 설정이나 권한 문제를 보면 되고, 인증이 안 되면 로컬 프로파일과 자격증명부터 점검하면 된다.
