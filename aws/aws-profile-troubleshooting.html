01 Config vs Credentials — 뭐가 다른가

AWS CLI는 두 개의 파일로 설정을 관리합니다. 혼용해서 쓰면 헷갈리니 역할을 먼저 구분하세요.

설정 파일
config

리전, 출력 형식, role_arn, source_profile 등 동작 방식을 정의합니다. 민감 정보는 없습니다.

~/.aws/config
인증 파일
credentials

Access Key ID, Secret Key, Session Token 등 실제 자격증명을 저장합니다. 외부 노출 금지.

~/.aws/credentials
Profile이란?

두 파일에서 [profile 이름] (config) / [이름] (credentials)으로 묶인 설정 블록입니다. --profile 플래그로 원하는 계정/역할로 전환할 수 있습니다.

02 파일 위치 및 확인 방법

파일 경로

경로
config      : C:\Users\사용자명\.aws\config
credentials : C:\Users\사용자명\.aws\credentials

터미널에서 내용 보기

cmd / PowerShell
# config 내용 출력
> type %USERPROFILE%\.aws\config

# credentials 내용 출력
> type %USERPROFILE%\.aws\credentials

# 메모장으로 열기
> notepad %USERPROFILE%\.aws\credentials

프로파일 목록 확인

cmd / PowerShell
> aws configure list-profiles
devops
user-ai-hyejun
default

# 특정 프로파일 설정 상세 보기
> aws configure list --profile devops

경로

03 주요 진단 명령어

명령어 설명
aws configure list-profiles 등록된 프로파일 전체 목록 출력
aws configure list --profile <name> 특정 프로파일의 설정(region, key, source 등) 확인
aws sts get-caller-identity --profile <name> 해당 프로파일로 실제 AWS 인증이 되는지 확인 — 가장 먼저 실행할 진단 명령
aws configure --profile <name> 프로파일 자격증명 등록 또는 갱신

04 실제 트러블슈팅 — 에러부터 해결까지

SSM 포트포워딩 시도 중 발생한 에러를 단계별로 추적한 흐름입니다.

에러 발생

cmd
> aws ssm start-session \
    --target i-0b3439f7b2088d79b \
    --document-name AWS-StartPortForwardingSession \
    --parameters "{\"portNumber\":[\"443\"],\"localPortNumber\":[\"8443\"]}" \
    --profile devops

aws: [ERROR]: An error occurred (InvalidClientTokenId)
when calling the AssumeRole operation:
The security token included in the request is invalid.

원인 추적 타임라인

Step 1 — 에러 확인
InvalidClientTokenId 발생

devops 프로파일로 AssumeRole 시도 중 토큰 무효 에러. 자격증명 문제로 일단 의심.

Step 2 — source profile 추적
devops는 default를 source_profile로 사용

config 확인 결과 devops는 role_arn을 AssumeRole 하되 source_profile = default. default가 문제일 가능성 확인.

Step 3 — default도 실패
aws sts get-caller-identity --profile default 에러

default 프로파일도 InvalidAccessKeyId — credentials 파일의 Access Key 자체가 AWS에 존재하지 않음.

Step 4 — 파일 직접 확인
credentials 파일 열어보니 [devops] 섹션 자체가 없음

핵심 원인 발견. [default]만 남아있고 [devops] 블록이 누락되어 있었음. 어느 시점에 파일이 초기화되거나 덮어써진 것으로 추정.

Step 5 — 해결
aws configure --profile devops 로 재등록

새 Access Key를 발급받아 devops 프로파일을 재등록. SSM 포트포워딩 정상 동작 확인.

05 credentials 파일 구조 이해

에러 유형에 따라 자격증명 파일 안에 어떤 항목이 있는지 확인하는 것이 진단의 핵심입니다.

~/.aws/credentials 예시
[default]
aws_access_key_id     = AKIA...          # AKIA로 시작 = 장기 키
aws_secret_access_key = xxxxxxxxxxxxxx

[devops]                                  # ← 이 블록이 없으면 프로파일 누락
aws_access_key_id     = AKIA...
aws_secret_access_key = xxxxxxxxxxxxxx
aws_session_token     = xxxxx...         # 있으면 임시 자격증명 → 만료 가능
에러 메시지 원인 해결
InvalidClientTokenId 자격증명 무효 (만료, 삭제, 누락) 프로파일 재등록 또는 새 키 발급
InvalidAccessKeyId 해당 Access Key가 AWS에 존재하지 않음 IAM 콘솔에서 새 키 발급
ExpiredTokenException aws_session_token 만료 임시 자격증명 재발급 (STS / SSO)
프로파일 없음 / NoCredentialProviders credentials 파일에 해당 블록 누락 aws configure --profile <name> 으로 재등록

주의

aws configure (프로파일 지정 없음) 를 실행하면 [default]만 갱신되고 다른 프로파일 블록은 그대로입니다. 단, credentials 파일이 어떤 이유로 전체 초기화되면 모든 프로파일이 사라지므로 주기적으로 파일을 백업해두세요.

07 예방 및 관리 팁

credentials 파일 백업

팀 내 스크립트나 Terraform 등이 aws configure를 자동 실행하면 파일이 덮어써질 수 있습니다. 프로파일 추가 전후로 파일을 별도로 복사해두세요.

source_profile 의존 관계 파악

config에서 source_profile = default로 설정된 프로파일은 default 자격증명도 유효해야 합니다. 에러 발생 시 source_profile 먼저 점검하세요.

aws_session_token 여부 확인

credentials에 aws_session_token 항목이 있으면 임시 자격증명입니다. 수 시간~수일 내 만료되므로 에러 발생 시 이 값부터 확인하세요.

진단 첫 번째 명령

어떤 에러든 먼저 aws sts get-caller-identity --profile <name> 부터 실행하세요. 인증이 되면 AWS 측 문제, 안 되면 로컬 자격증명 문제입니다.
