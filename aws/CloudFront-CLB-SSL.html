AWS SSL / TLS Multi-Region 장애 분석

AWS 멀티리전 환경에서
CloudFront + CLB SSL 인증서 구성과 장애 분석

CloudFront, Classic Load Balancer, EC2로 구성된 멀티리전 아키텍처에서 SSL 인증서를 어느 계층에 배포해야 하는지, 그리고 잘못된 구성이 어떤 장애로 이어지는지를 실제 사례를 통해 정리한다.

버지니아 북부 · 아일랜드 · 싱가포르 · 서울 · CloudFront · Classic Load Balancer · EC2

01 아키텍처 구성

  • CloudFront 배포: 총 4개 — 운영 3개 (버지니아 북부, 아일랜드, 싱가포르) + 스테이지 1개 (서울)
  • 오리진: 각 CloudFront는 동일 리전의 CLB를 오리진으로 사용
  • EC2: 각 CLB 하위에 인스턴스 2대 구성 (WEB/WAS 통합)
  • 인증서: 외부 CA에서 발급받은 인증서 파일을 ACM에 임포트하여 사용

04 인증서 만료로 인한 장애 분석

항목내용
증상특정 리전 웹 페이지 5xx 에러 발생. SSH 서버 접속은 정상.
범위버지니아 북부만 정상, 아일랜드·싱가포르·서울 접속 불가
원인CloudFront 오리진 프로토콜이 HTTPS였으나, 일부 리전 CLB에 유효한 인증서가 적용되지 않아 SSL/TLS 핸드셰이크 실패 (추정)
해결전 리전 CLB에 유효한 인증서 배포

원인 추적 타임라인

Step 1 — 에러 확인
특정 리전 5xx 에러 발생

서버 SSH 접속은 정상이나 웹 페이지만 접속 불가. 인증서 만료 기간과 시점 일치.

Step 2 — 범위 확인
버지니아 리전만 정상, 나머지 3개 리전 장애

버지니아 CLB에만 유효한 인증서가 남아있었고, 다른 리전 CLB는 인증서가 만료되었거나 갱신이 반영되지 않은 상태.

Step 3 — 근본 원인
CloudFront → CLB 구간 SSL 핸드셰이크 실패

CloudFront 오리진 프로토콜이 HTTPS로 설정되어 있어, CLB에 유효한 인증서가 없으면 연결 자체가 실패한다.

Step 4 — 해결
전 리전 CLB에 유효한 인증서 배포 후 정상화

동일 인증서 파일을 4개 리전 ACM에 각각 임포트하고 CLB에 적용. 전 리전 정상 서비스 확인.

05 올바른 인증서 배포 전략

CloudFront 인증서 필수 조건

CloudFront에 적용되는 인증서는 반드시 ACM us-east-1 (버지니아 북부) 리전에 등록되어 있어야 한다. CloudFront는 글로벌 서비스이지만 인증서는 us-east-1에서만 참조한다.

동일한 인증서 파일을 기준으로, CloudFront용 인증서는 ACM us-east-1에 등록하여 각 CloudFront 배포에 연결하고, CLB용 인증서는 각 CLB가 위치한 리전의 ACM에 등록하여 해당 CLB에 연결한다. 인증서 파일 자체는 동일하더라도 사용 대상에 따라 ACM 등록 리전이 달라진다.

인증서 배포 맵

ACM 임포트 4번 → 총 8개 리소스 참조
외부 CA 발급 인증서 파일 (1개)
│
├── ACM us-east-1 임포트
│     ├── CloudFront 배포 — 버지니아 북부 Origin 참조
│     ├── CloudFront 배포 — 아일랜드 Origin 참조
│     ├── CloudFront 배포 — 싱가포르 Origin 참조
│     ├── CloudFront 배포 — 서울/스테이지 Origin 참조
│     └── CLB (버지니아 북부)        참조
│
├── ACM eu-west-1 임포트
│     └── CLB (아일랜드)             참조
│
├── ACM ap-southeast-1 임포트
│     └── CLB (싱가포르)             참조
│

eu-west-1 (아일랜드)아일랜드 CLB

핵심 요약

인증서 파일은 1개. ACM 임포트는 4번 (리전 수만큼). CloudFront 배포 4개는 각각 다른 리전의 Origin을 바라보지만, CloudFront 자체는 글로벌 서비스이므로 인증서는 모두 ACM us-east-1에 등록된 동일 인증서를 사용할 수 있다.

06 정리

Point 01
SSL 종료 지점을 파악하라

CLB 리스너의 인스턴스 프로토콜이 HTTP이면 SSL은 CLB에서 종료된다. EC2에는 인증서가 불필요하다.

Point 02
CLB에도 인증서가 필요하다

CloudFront → CLB 구간은 HTTPS 오리진 프로토콜. CLB 인증서 만료/미적용 = 서비스 장애.

Point 03
CloudFront는 us-east-1만

CloudFront 배포 4개는 ACM us-east-1에 등록된 동일 인증서를 참조할 수 있다. 각 배포에 별도 인증서 임포트는 필요하지 않다.

Point 04
라우팅 패턴별 전략이 다르다

EC2 직접 연결 도메인은 HTTPS 접근 시 WAS 인증서가 필요하다. Route53 레코드를 먼저 확인하자.
